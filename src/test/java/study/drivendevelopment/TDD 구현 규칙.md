### TDD 공통 규칙

### 

1. 현재 파일에서 명시된 항목을 순차적으로 진행한다.
2. 제시된 기능 요구사항의 파일을 읽는다.
3. 기능 요구사항이 순차적으로 구분되어 있다면, 4~8번의 항목을 각각 요구사항에 적용한다.
4. 테스트 코드의 형식은 Given-When-Then 구조를 맞춰서 작성하도록 한다.
5. 테스트를 위한 메소드 작성 규칙은 아래와 같다.
    1. 메소드명은 한글로 작성한다.
        1. `전화번호_검증_성공_테스트` , `배송_주소_null_예외_테스트`
    2. 파라미터가 필요한 경우 명시해준다.
        1. `@ParameterizedTest` , `@ValueSource`
    3. 테스트 기능의 항목은 순차적으로 적어준다.
        1. `@DisplayName`
6. TDD의 개발 주기 Red-Green-Refactor를 따른다.
    1. **Red (실패):** 구현하려고 하는 기능의 테스트 케이스를 먼저 작성한다. 실제 코드의 유무부터 판별이 진행되도록 한다.
    2. **Green (통과):** 실패 했을 때 통과할 수 있는 경우의 코드를 작성한다.
    3. **Refactor (리팩토링):** 기능은 유지하되, 중복 코드 혹은 불필요 부분을 재구성한다. 반드시 기존에 통과된 기능은 유지되어야 한다.
        1. **테스트는 public으로 진행하되, 기능 검증이 완료된 후 기능 요구사항에 접근제한자 명시가 없으면 private으로 변경한다.**
7. 5번의 항목이 완료 시 새로운 md 파일로 정리를 한다.
    1. 파일명은 클래스 이름 + 기능 요구사항 테스트 결과 형식으로 만든다.
    2. 테스트 결과물을 표로 정리 하며, 각 열의 항목은 아래와 같다.
        1. 기능 요구사항 이름 | 기능 요구사항 설명 | 테스트 세부 항목 | 수행 데이터 | 예상 결과 | 실패 사유 | 통과 데이터
        2. 요구사항 설명은 검증이 되는 내용을 중심으로 작성한다.
    3. 각 기능 테스트가 통과되면 7-b 항목을 바로 적용해준다.
8. 4~7번 사항을 반복한다.
9. 기능 요구사항 파일을 모두 수행할 경우 마친다.
